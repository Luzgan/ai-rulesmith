# Comments Explain Why Not What

- Don't comment obvious code — good names and structure are self-documenting
- Do comment non-obvious design decisions (why, not what)
- Comment when processing order matters or a fall-through is intentional
- Use TODO/FIXME sparingly and with context (what needs doing and why)
- Delete commented-out code — version control is the history
